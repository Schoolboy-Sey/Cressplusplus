CRESS++ : DESIGN DIRECTION & ONBOARDING MANIFESTOWelcome to the Cress++ engine.If you are joining this project, you need to leave traditional Object-Oriented Programming (OOP) at the door. We are not building a standard indie game; we are building a macroscopic, deterministic simulation capable of calculating millions of cellular automata interactions and thousands of kinetic entity collisions in a fraction of a millisecond.To achieve this scale, we worship the CPU's L1 Cache.This document outlines the core philosophies, architectural rules, and design choices that govern every line of code written for Cress++. If a feature cannot be implemented within these constraints, the feature is redesigned, not the architecture.1. THE CORE PHILOSOPHY: DATA-ORIENTED DESIGN (DOD)The primary bottleneck in modern computing is not math; it is memory fetching. If the CPU has to wait for data from the RAM stick, the framerate dies.The Rule of Flat Data: We do not use class Tile or class Unit. We do not use pointers, virtual functions, or deep inheritance trees. Everything is built using strictly packed, 1-Dimensional arrays (Structure of Arrays / SoA).The 16-Byte Reality: The physical world is exactly 16 bytes per coordinate. Do not add to it. By keeping the Tile struct exactly 16 bytes, the CPU can pull exactly 4 tiles per 64-byte Cache Line, guaranteeing perfect memory streaming.Entities are Indices: An entity is just an integer ID (e.g., 42). To find its weight, you read Entity_BaseWeight[42]. To find its team, you read Entity_Team[42]. This guarantees contiguous memory access during bulk operations.2. BRANCHLESS DETERMINISM (The Math of the Universe)We do not use complex if/else chains to determine how the world reacts. Branches stall the CPU pipeline. We use Branchless Mathematics, Lookup Tables (LUTs), and Bitwise Logic.The 64-Bit Stack: Transient effects (Fire, Water, Poison) are bits in a 64-bit integer. We use hardware intrinsic functions (__builtin_ctzll) to instantly jump to active bits, skipping empty space.LUTs over Logic: How does Fire know what to burn? It doesn't use if (biome == GRASS). It uses an $O(1)$ array lookup: flammability_lut[neighbor.composition].Complete Determinism: The physics engine does not use Random Number Generators (RNG). If the same actions are inputted on the same frame, the exact same chain reaction of chemistry, explosions, and fluid flow will occur 100% of the time.3. MACRO-SCALE SYSTEMS ($O(1)$ and SIMD)Because we are simulating thousands of units and millions of tiles, we cannot use algorithms that scale exponentially.No $O(N^2)$ Collision: Entities never check the coordinates of other entities. We maintain an Occupancy_Grid. To see if a unit hits an enemy, it reads Occupancy_Grid[target_pos]. It takes 1 clock cycle.SIMD Biology (AVX2): The biological ecosystem (plant growth, animal scent tracking) uses the 16-bit imprint_field. We process this using AVX2 intrinsic headers, calculating the cellular automata for 16 tiles simultaneously in a single CPU instruction. We pad our local chunk memory (e.g., 48x34) explicitly to ensure memory addresses perfectly align with the 32-byte SIMD registers.4. THE TIMELINE & THRESHOLD COMBAT (Game Design Rules)Cress++ is a game of proactive timeline manipulation, not reactive guessing.WeGo Execution: Turns are divided into discrete "Steps". Actions do not happen instantly; they are placed on the timeline.Enemies Play First: The AI calculates and locks in its pathing/attacks before the player acts. The player's job is to inject "Instants" and spells into the timeline to disrupt the enemy's guaranteed future.Mass over HP: We do not use Health Points. Combat is a kinetic physics puzzle calculated as BaseWeight + Velocity = Force. If a 50-weight unit hits a 100-weight boss, the 50-weight unit shatters. To win, the player must use the environment (e.g., Acid, Poison) to mathematically degrade the boss's effective weight before the strike connects.The "Wile E. Coyote" Window: Gravity and hazards do not kill instantly. If a bridge is blown out from under a unit, the unit enters a hovering state until the next execution step, giving players a 1-step tactical window to save them.5. THE FRONT-END CONTRACT (Godot's Role)Godot is the monitor; C++ is the brain.Zero Math in GDScript: Godot does not calculate pathfinding, chemistry, or combat. Godot is only allowed to capture user input, pass it to the C++ GDExtension, and draw the results.Single Draw Calls: We do not spawn 1,000,000 Node3D blocks for the ground. The C++ passes the raw byte array to a Godot Vulkan Shader, which draws the entire map using a single MultiMeshInstance3D.Optical Illusions over Geometry: The physical mesh of the map is perfectly flat. The Godot shader reads the elevation byte from the C++ array and artificially draws "cliff faces" on the textures to create 3D verticality.Onboarding Checklist for New Programmers:Understand Bitwise Math: You must be comfortable with bit-shifting (<<, >>), masks (&, |, ^), and two's complement.Learn the Compiler Intrinsics: Familiarize yourself with __builtin_popcount (used for evolution tiers) and _mm256_load_si256 (used for the biological SIMD loops).Respect the Sentinel: Remember that unsigned integers cannot hold -1. An empty tile in the occupancy grid is EMPTY_TILE (65535).No Dynamic Allocation in the Loop: Never use new, malloc, or dynamically resizing containers (like standard std::queue) during the run_step() physics loop. Pre-allocate your buffers.We are building a clockwork universe. Keep the data flat, keep the instructions branchless, and let the processor do what it was built to do.
